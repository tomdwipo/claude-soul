# A shortcut/intercept must gate on READINESS, not just on the trigger

**When this applies:** there's a fast path / intercept / guard clause / cache / router that **short-circuits before the full pipeline** (before context is assembled, before the LLM, before extraction). Any time a shortcut can reply or act **earlier** than the normal path.

**Principle:** a shortcut that pre-empts the pipeline must gate on **two** conditions — (1) the trigger matches, **AND** (2) it actually **has what it needs** to act correctly. A trigger match alone is not enough. If the shortcut can return a **template response** (e.g. "please provide X") just because the trigger fired, it **throws away context the user may already have provided** — because it cuts the path *before* the part that would read that context. The correct pattern is **try to fulfil first, then ask** — fulfil-first, ask-only-on-gap.

**Why:** consider an intercept that runs at the very front of the turn handler, before recent context is gathered. In its naive form it does "if intent regex matches, return the opener" — replying with a template ("what field? what location?") on every regex hit, **without checking whether the needed slots are already filled**. The result: a user who already typed their full criteria (plus an attached document) gets **asked again** for what they just answered. Meanwhile the extractor that reads the transcript and the attachment **already exists** — it just needs to be called first. The fix: make the front path **extract-first** → check a readiness predicate → act immediately if there's enough, fall back to the opener only when something's missing. The contrasting evidence: when the intercept is accidentally skipped (a typo that doesn't match the regex), the full path reads the attachment and works correctly — proving the context was there all along; the shortcut was discarding it.

**How to apply:**
- Structure the shortcut as **fulfil-first**: call the fulfilment step (extract/lookup/compute) **first**, then decide. The "ask for info" / opener / fallback response is the **last** branch, not the default the moment the trigger hits.
- Gate the action on **sufficiency**, not trigger-presence. Make the readiness check **explicit and pure** (e.g. a `slots_ready(spec)` function) so it can be unit-tested deterministically without the LLM.
- **Fallback = the old behavior (degrade-safe):** if the fulfilment step fails → fall back to the previous path, not to a wrong action. An extractor failure that returns to the opener is identical to pre-fix behavior.
- Be wary of shortcuts placed **before** context assembly: that's the spot most prone to discarding context. If it genuinely must sit up front (for cost/latency reasons), pay for it with a readiness check, not a blind template.
- **Danger sign in logs/telemetry:** a **verbatim-identical** template response appearing across turns/users with different context → almost certainly a blind shortcut, not context-aware output.

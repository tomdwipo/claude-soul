# A security guarantee lives in the structural capability boundary (independently verified), not in prompt refusal — the prompt-gate is a soft outer layer, isolation is the backstop

**When this applies:** someone asks "can a user abuse this?" / "is it safe from exfiltration or destructive actions?" about a system that runs an LLM over user input (a chat agent, tool-use, a sandbox). You're about to answer or guarantee that something malicious (reading secrets, accessing another user's data, reaching the host, causing damage) cannot happen.

**Principle:** prompt refusal is **probabilistic** — the model is cooperative but promptable, and an attacker has infinite phrasings, so eventually one slips through *in how it's asked*. So never treat prompt refusal as the security guarantee. The **hard** guarantee lives in the **structural capability boundary** — what the process can **physically reach**, regardless of whether the model chooses to comply:
- **Network:** if egress is blocked, the process can't reach the host or the internet → the exfiltration channel is dead even if the model is coerced into sending.
- **Mounts:** if secret files / the database / the host filesystem aren't mounted, there's nothing to read, period.
- **Privilege:** read-only root filesystem + drop all capabilities + non-root + no-new-privileges → can't damage or escalate.
- **Authz scope:** the credential carried is the narrowest-scoped token (user-scoped, not an admin/operator key) → can't touch other users' data.
- **Secret placement:** the real key lives in a trusted component (e.g. a proxy); the token the sandbox holds is gateway-only and useless outside that network.

Defense-in-depth: the prompt-gate is a soft outer layer (it reduces attempts and keeps UX polite); isolation is the backstop that holds even if the outer layer leaks. Prove **both**, but anchor the "cannot happen" claim to the structural one.

**Why:** after fixing a scope gate, someone asks "users can't abuse this, right?" The tempting move is to answer "yes, it's safe" on the strength of a green prompt eval. But a prompt eval is the soft layer — probabilistic, with a tiny sample size. The honest move is to verify the capability layer against the **actual production config**: confirm egress is blocked, only a per-user work directory is mounted (no secrets, no database), the process runs read-only with capabilities dropped as a non-root user, the credential carried is a user-scoped token (admin actions require a separate admin key), and the real API key lives behind the proxy. Only then red-team the prompt layer across many attack categories (jailbreak/roleplay, other-user PII, env/secret exfiltration, reading system credential files, wiping the database, recursive delete, injected admin-override) and confirm every one is refused with zero leaks. The honest answer becomes "no — not because the agent is polite, but because the wall is structural," plus the caveat that the prompt layer is probabilistic but backstopped by isolation. Lean only on the prompt eval and the "safe" claim is brittle — the next bypass phrasing breaks it.

**How to apply:**
- Asked "is it safe / can it be abused?" → don't answer from a prompt eval alone. Map the capability boundary first: network reachability, mounts, privilege/capabilities, the authz scope of carried credentials, secret placement. That is the source of the guarantee.
- **Verify the boundary against the actual production config**, not by reading code or docs blindly — check the real runtime: is egress blocked? what's mounted? are capabilities dropped? what's the token scope? (see [[validate-in-real-runtime-context]] and [[current-state-overrides-stale-memory]]).
- Stress-test the prompt layer (multi-category red team: jailbreak, data exfiltration, secret exfiltration, destructive actions, injected override) to measure the **outer** layer — but treat results as probabilistic (a small n is not proof), and remember automated detection can miss (read the actual reply text, don't just keyword-flag).
- Anchor the "cannot happen" claim to the structural boundary and state the soft caveat honestly. If a structural backstop is absent (e.g. a secret is mounted, or network egress is open), that's a **serious** finding — fix the isolation, don't patch the prompt.

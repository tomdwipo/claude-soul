# Threat-model a feature across EVERY OWASP family that matches its surfaces (LLM + Web + API), not just the LLM Top 10 — at design time

**When this applies:** you're designing or reviewing a feature that spans **more than one surface** at once — e.g. an LLM agent exposed via a **web endpoint + API** (a chat widget, a webhook, a REST route that calls an LLM, a public tool/MCP route). The moment someone asks "is it safe?" or you write a Threat Model section.

**Principle:** OWASP maintains **several different Top 10 families, one per surface type** — an **LLM Top 10** (LLM apps), a **Web Top 10** (web apps), an **API Security Top 10** (API endpoints). One feature often touches **more than one**. Threat-modeling with the LLM Top 10 alone is **blind** to web/API-native risks that aren't on the LLM list: **broken object-level authorization (BOLA/IDOR)** — cross-user object access; **SSRF** as a web risk; **security misconfiguration** (leaked errors/traces, CORS); **vulnerable components** (dependency CVEs); **auth failures**. So: **map the feature's surfaces first → pull in EVERY relevant OWASP family → merge into one Threat Model** (sub-tables per lens are fine). Still anchor the **hard** guarantee to the [[security-guarantee-lives-in-capability-boundary-not-prompt]] capability boundary, not the prompt; many web items (SSRF, command injection) are **backstopped by structural isolation** when present — mark which are backstopped vs. which need a new guard. Do this at **design time** (the cheapest moment), not in a post-incident audit ([[untrusted-text-into-prompt-is-injection-surface]]).

**Why:** consider designing an anonymous web-chat channel. The initial Threat Model covers only the OWASP **LLM** Top 10 (XSS, excessive agency, unbounded consumption, prompt injection). A reviewer asks "besides the LLM Top 10, what else? there's an OWASP Top 10 too, right?" — and that's correct: the LLM lens **misses** purely web/API items. Adding the Web + API lenses surfaces new gaps the LLM Top 10 never catches: **BOLA/IDOR** (the chat endpoint must resolve the user identity **server-side** from the session token and must **not** accept a user id from the request body — otherwise one person's session is reachable by another), **security misconfiguration** (don't leak stack traces to the client), **vulnerable dependencies** (pin + audit). Meanwhile several other web items (SSRF via a fetch-URL skill, command injection) turn out to be **already backstopped by the sandbox** (no egress, no host filesystem) — so the second lens not only finds new risks but also **confirms which are already structurally safe**. Stop at the LLM Top 10 and BOLA + misconfig ship to production.

**How to apply:**
- **Map the surfaces first:** is this feature web? API? LLM-backed? sandbox/tool-use? → each "yes" pulls in one OWASP family. (web → Web Top 10, API → API Security Top 10, LLM → LLM Top 10, ML → MITRE ATLAS if needed.)
- **One Threat Model, sub-tables per lens** in the design doc: columns `risk → mitigation (in code/boundary) → phase`. Mark **overlapping** items once (e.g. XSS appears under both LLM and Web), and mark items **already backstopped** (SSRF/cmd-injection ← isolation) so focus stays on what's genuinely new.
- **Web/API items most often missed by an LLM-only lens:** BOLA/IDOR (per-owner object scope; resolve identity server-side, not from input), SSRF (allowlist / egress backstop), security misconfig (generic errors, CORS, headers), vulnerable components (pin + scan), auth failures (rate-limit, token entropy, single-use codes with a TTL).
- **Hard guarantee → the capability boundary**, verified against the actual production config; the prompt is the soft layer. For a feature that reuses an existing engine, **state the reuse** (same boundary → guarantee inherited) — don't re-derive.
- **Eval cases in the implementation plan**: transcripts/requests that try each category (inject, exfiltrate, access another session/IDOR, invoke an expensive capability) → must be refused or gated **deterministically**.

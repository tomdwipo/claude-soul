# Measure the agent, don't just coach it (eval engineering)
**When this applies:** writing a principle/rule to "teach" an agent, or deciding whether a learning actually stuck.
**Principle:** every behaviour you want from an agent needs a *checkable signal*, not only prose. A principle without an eval case is a hope, not a guarantee — you'll *feel* the agent improved without being able to *prove* it.
**Why:** the `.docs/` loop already teaches judgment (input-side). The missing half is output-side: how do you know it worked? Galileo's "eval engineering" names the gap — trusting the agent requires first trusting the eval. Their numbers: a raw LLM-as-judge agrees with a human only ~66-70% of the time; with a subject-matter expert tuning the prompt + a few edge-case examples it jumps to 95%+. The lever is the *expert + concrete examples*, not a bigger model.

> **Grounding (via a paper search + reading the source).** The ~66–70% lower bound is corroborated peer-reviewed: **MT-Bench** (Zheng et al. 2023, `arXiv:2306.05685`, Tables 5–6, read directly) reports GPT-4-judge↔human agreement of **66% under setup S1** (includes ties/inconsistencies), rising to **85% under S2** (non-tie votes only), vs **81% human↔human**. So "raw agreement" is really **66–85% depending on the tie/consistency setup**, and a strong LLM judge can *match human–human consistency*. The **"95%+ with SME tuning"** figure is a **Galileo vendor claim, not independently verified** — treat as directional, not fact. The durable takeaway is the **mechanism** (expert + concrete examples beats a bigger model), not the exact number. (Process applied here = [[ground-empirical-claims-not-judgment-in-literature]] #54.)
**How to apply:**
- When a learning is **settled by build/test** → it already has a signal; catalog it in `.docs/common-issues/` as usual.
- When a learning is a **judgment call** (a principle) → attach a tiny eval case: *1 trigger + expected behaviour*. That mini-eval is what lets a future session (or you) replay it as a regression check.
- **You are the SME-in-the-loop.** Your correction is the ground truth; `/learn` is the tuning step; the failure modes land in `.docs/common-issues/`. Recycle: a new failure in the wild → refine the principle → log in `recent-updates`. This is the same lifecycle, just named.
- **Don't over-build.** Skip Galileo's heavy infra (SLM fine-tune, runtime guardrails) at workspace scale — adopt the *mental model* (adaptive, expert-in-loop, eval→guardrail), not the machinery. Demote a principle to a hard rule only once a real check can settle it ([[lessons-to-catalog]]).
- For a **product** that ships agents to end users, the same loop becomes a moat: cheap fine-tuned small-model judges on 100% of traffic (not 10% sampling) → lower cost per interaction + measurable quality. See [[catalog-lookup-every-first]] for the input-side twin of this output-side discipline.

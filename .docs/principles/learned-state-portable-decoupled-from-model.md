# Keep learned/adaptive state as portable DATA, decoupled from the model that consumes it

**When this applies:** you're building any system that *learns* or *adapts* per-user/per-tenant over time — preference memory, a "soul", personalization, RAG context, a feedback loop — and the obvious place for "what we learned" is some property of the model (a fine-tune, a per-user adapter, weights). You also pick an LLM vendor that you might swap later (cost, quality, ToS, availability).

**Principle:** store the learned state as **plain, model-independent data** (text rows in your DB) and **inject** it into the model at call time, instead of baking it into the model's weights. Then the learning asset lives in *your* store keyed by tenant id, not in a vendor's model. Swapping the model — across tiers or to a different vendor entirely — carries the accumulated state over **instantly at the next turn**, with zero re-learning, because the model only ever *reads* the same text. Coupling "what we learned" to a specific model is a single-vendor bet you pay for again every time you switch; decoupled data is portable and lowers lock-in. The one place coupling is unavoidable — an **embedder** used for dedup/recall — should be isolated and treated as load-bearing config, never swapped casually.

**Why:** consider a per-user "soul". Learned style preferences are distilled to **text** in a preferences table and re-injected into the system prompt every turn. The three pipeline components are deliberately decoupled: the **chat model** (consumes the state), the **distiller** (produces it, can be a cheap separate model), and the **embedder** (a small local model for dedup) are independent — and the stored preference is just a string. So changing the chat model needs **no migration and no warm-up**: the same rows feed the new model on turn one. That portability *is* the moat — memory that carries over is the durable advantage — and it kills vendor lock-in for the expensive component. The subtle trap: only the **embedder** must stay fixed — change it and the stored vectors are no longer comparable to fresh ones, so dedup/similarity (cosine >= threshold) silently mis-merges or duplicates. Note also: a counter that promotes/retires text (a "reinforcement count") is *adaptive memory*, **not** reinforcement learning — no reward, no gradient, weights frozen. Call it what it is so the claim survives a skeptical engineer.

**How to apply:**
- Put "what the system learned" in a **portable store as data** (text/JSON rows), keyed by the tenant; have the model *read* it at call time. Don't encode it as weights/fine-tune unless portability genuinely doesn't matter.
- Keep the **adapting components decoupled from the swappable one**: the model that consumes the state, the one that produces it, and any embedder should each be independently configurable.
- Treat the **embedder as the real coupling**: pin it, document that changing it invalidates stored vectors, and plan a re-embed migration if you ever must change it — unlike the chat model, it is *not* free to swap.
- Be honest in naming: "preference/adaptive memory, portable across models" — not "per-user RL" — when a counter (not a reward+gradient) is doing the work. Overclaiming invites a correct rebuttal.
- Pairs with [[dont-expose-the-recipe]] (learned data is internal) and [[validate-in-real-runtime-context]] (verify which model actually reads the state in the real runtime path, not a host probe).

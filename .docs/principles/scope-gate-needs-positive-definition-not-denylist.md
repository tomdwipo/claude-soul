# A scope/safety gate by enumerated examples only covers what you listed — domain-refusal needs a POSITIVE scope definition plus weight that beats the "you can do anything" framing

**When this applies:** writing a refusal gate (scope/safety/abuse) in the system prompt of a cooperative model — "refuse if out of scope / if intent is abuse". Deciding how to phrase it so it actually holds, instead of merely appearing to be there.

**Principle:** the model pattern-matches to the **examples you wrote**, not to the abstract category those examples were meant to represent. A gate written as an **enumerated denylist** ("refuse hacking, mining, spam, scraping") only covers the enumerated items — a request that's out of domain but off the list (e.g. trading signals, medical/legal advice) slips through, because the model reasons "is this hacking/spam? no → allowed". Two conditions for a domain-refusal gate to actually hold:
1. **A POSITIVE scope definition**, not just a negative denylist. State what is IN scope ("you only help with: A, B, C") as the contrast, plus a few representative out-of-domain examples (trading/financial, medical, legal) — so the model has a category line, not a per-item guess.
2. **Gate weight ≥ competing instructions.** The gate is in a tug-of-war with other instructions in the same prompt. If the prompt repeatedly says "you're an agent WITH tools, you CAN do all this, help PROACTIVELY, don't be passive", one soft line of "refuse what's out of scope" will lose → the model chooses to help. The gate's strength and specificity must match the "you can" framing it fights (a deflect reflex as firm as the identity gate that already works).

**Why:** A cooperative agent in production emitted out-of-domain financial trading signals to a user — outside its real domain, at many times the normal per-turn cost, with financial-advice risk. Forensics: the model's base layer fabricated it (not a skill — the user had no such skill), and the scope gate WAS present and active in prod (verified, not overridden away). It was bypassed because of exactly the two flaws above: its examples were all security-abuse cases (hacking/mining/spam/scraping) with zero domain-creep examples; and one soft line lost against repeated "you CAN do everything" plus "be proactive, don't be passive" framing. Contrast: the IDENTITY gate ("don't reveal the model/vendor") DID hold — because it was firm, specific, and supplied a replacement behavior (deflect, then pivot). The scope gate didn't get the same treatment.

**How to apply:**
- Writing a refusal gate → **enumeration = illustration, NOT definition.** Always pair it with a positive category definition (what's in scope) so the model has a line, not a checklist.
- For domain scope, **derive the in-scope set from the registry/catalog** ([[derive-agent-capabilities-from-registry]]) — not just a list of bans. "You help with {capability list}; outside that, deflect" beats "don't go off-topic".
- **Audit the tug-of-war:** read the whole prompt, count instructions pushing "help/can/proactive" vs. those restraining. If the gate is outnumbered or out-firmed, strengthen it (or give a concrete replacement behavior: "deflect, then pivot to what you can do") — don't rely on one line.
- The mechanism question (instruction vs. deterministic) is answered by [[control-follows-model-disposition]] — a cooperative model → instructions suffice. This principle is the next layer: if you choose instructions, the **specificity** (positive vs. denylist) and **weight** determine what actually holds. For triggers where the model resists, see [[intercept-deterministically-when-model-resists]].
- Verify with an eval: off-domain cases (trading/medical/legal) must deflect, in-domain must be helped — so you neither over-refuse nor under-refuse.

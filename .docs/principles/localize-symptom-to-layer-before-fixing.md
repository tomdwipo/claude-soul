# Localize a "missing/wrong" symptom to its LAYER before designing a fix

**When this applies:** a report of a visual/behavioral symptom — "my data disappears on refresh", "results show up doubled", "the content won't appear", "the button does nothing" — that could originate in **more than one layer** (persistence → restore/API → client render → scroll/viewport/CSS). The moment you're about to start planning or designing a fix for a symptom like this.

**Principle:** **Don't design a fix until you've pinned the symptom to ONE layer by checking the REAL output of each layer.** The same symptom ("gone after refresh") can be born in different layers: data not saved (persistence), saved but the API doesn't return it (restore), API returns it but the frontend doesn't render it (render), or rendered but scrolled off / covered (viewport/CSS). Each layer has its **own cheap, deterministic probe** — query the store, `curl` the actual endpoint, inspect the payload, check the CSS positioning. Run the probes bottom-up until you find the layer where reality ≠ expectation. Fixing the wrong layer (e.g. rewriting the restore path when the real issue is render clutter) is effort plus regression risk spent on a problem that doesn't exist.

**Why:** A user reports "my content disappears on refresh — is that part of this fix?" The tempting first guess (almost written into the design) is a restore bug or a deploy lag. Per-layer probes refute it: querying the store shows the record WAS saved; curling the real history endpoint returns the full list including the supposedly-missing items, and the deployed code matches HEAD (so not a deploy lag); what's left is render clutter (stale results re-rendered on top of each other) plus a layout rule that keeps the input bar always visible — so "input visible" ≠ "scrolled to the bottom". Conclusion: not data loss / not backend — an illusion of render plus scroll. Trusting the symptom directly would mean rewriting an already-correct restore path and hunting a "restore bug" that isn't there. (A sibling lesson: a flag/column with single-writer semantics — written by one path and read by downstream consumers — must not be quietly co-opted by a second writer; that pollutes the signal. Verify a column's meaning in the data-model doc before reusing it.)

**How to apply:**
- Map the layers that COULD be the source (persistence → API → render → viewport/CSS). Each layer = one cheap probe: a store query, a `curl` of the endpoint the client actually calls, inspecting the payload, checking the CSS.
- Run probes **bottom-up**, stopping at the first layer where reality ≠ expectation. That's the layer to fix — not the ones below (already proven correct) or above (mere effects).
- **Curl the actual endpoint** rather than reasoning "the newest N should include it" — correct code ≠ correct output when another layer is in play. Verify, don't trust ([[current-state-overrides-stale-memory]]).
- Beware viewport illusions: sticky/fixed elements make "X is visible" ≠ "scroll is at X". Check the layout CSS before concluding "content is missing".
- If probes show the lower layers are correct, write it explicitly in the design ("backend curl-verified to return the full set; symptom = render/scroll layer") so downstream implementation doesn't chase a phantom. Complements [[smart-detect-done]].

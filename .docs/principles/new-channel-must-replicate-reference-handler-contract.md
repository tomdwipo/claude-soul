# A new channel that reuses the core engine MUST replicate the reference handler's contract — not just call the engine

**When this applies:** adding a new channel/surface (web chat, a messaging integration, a public API, etc.) that **reuses the core engine** (e.g. a turn-runner) from an existing channel. The moment you're about to design or implement it as "just call the engine".

**Principle:** the core engine only produces **raw output**. The reference channel wraps a **handler contract** around it: consuming signals/sentinels in the output (internal markers → enqueue work + strip them before display), delivering async results (notifications), reflecting state transitions in the UI (logged-in → hide the signup CTA), greeting/anti-hallucination guards. **Reusing the engine does NOT inherit that contract** — a new channel starts with an EMPTY handler. If it just does `return engine(...)`, you'll **leak internal signals**, **fail to start the queued work**, **fail to deliver results**, or **display stale state**. Before saying "just call the engine", **audit the reference channel's handler line by line** (what it does BEFORE and AFTER calling the engine), then replicate each relevant contract — ideally **lift it into a shared module** so the channels don't drift.

**Why:** A new channel reused the turn-runner but missed the reference channel's post-processing, producing a chain of bugs with the same root cause. (1) Internal markers **leaked** to the user because the new handler returned the raw reply, while the reference channel strips and enqueues them. (2) Async results were delivered only over the original channel, so the new channel's users got nothing until a polling endpoint was added. (3) The signup CTA didn't disappear after login because hiding was wired only into the reply turn, not the actual transition point. All three reduce to "engine called, handler contract not replicated". The fix lifted the sentinel parse/strip/enqueue logic into a shared runtime module so both channels share one source and can't drift.

**How to apply:**
- Before adding a channel: **read the reference channel's handler in full** — list every pre-engine and post-engine step (parse sentinels, enqueue, strip, notify/deliver, UI state update, guards).
- For each step, decide **replicate / share / intentionally skip** (and write down why). Nothing gets silently skipped.
- **Lift shared logic into a shared module** (next to where the engine emits sentinels) rather than copy-pasting — so the two channels can't diverge.
- Internal signals/sentinels emitted by the engine are a **contract**, not free text → every channel MUST consume and strip them before displaying (output-leak risk).
- Reflect **state transitions** (anon→login, quota change) at the **transition point itself**, not on the next event. Complements [[ground-the-gap-before-designing-a-missing-feature]], [[validate-in-real-runtime-context]], [[security-guarantee-lives-in-capability-boundary-not-prompt]].
- **Same move across a different kind of multi-path system:** [[feature-must-land-on-the-prod-path]] — there the paths are *flag-branches*; here they're *channels*. Both: lift shared behavior into ONE module every path assembles from, or paths silently diverge.
